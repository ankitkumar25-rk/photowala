PHOTOWALA - PREMIUM PHOTO & CUSTOM PRINTING PLATFORM
PROJECT ARCHITECTURE & KNOWLEDGE BASE FOR AI MODELS
================================================================================

1. PROJECT SUMMARY
------------------
Photowala is a high-end e-commerce platform specializing in premium gifts, trophies, 
momentos, and professional custom printing services. The platform is architected for 
scalability, security, and a premium "luxury" user experience.

The ecosystem is divided into three core modules:
- Backend: Express/Node.js REST API with Prisma ORM and PostgreSQL.
- Frontend Store: Customer-facing React SPA (Vite-powered).
- Frontend Admin: Administrative dashboard for order management and business analytics.

2. TECH STACK & ECOSYSTEM
-------------------------
- Core: Node.js (v22.12.0), Express, React 18, Vite.
- Styling: TailwindCSS with a "Mobile-First" philosophy and premium design tokens.
- State Management: 
    - Frontend: Zustand (Atomic stores for Auth, Cart, UI state).
    - Data Fetching: TanStack Query (React Query) for robust caching and synchronization.
- Security: 
    - Authentication: PASETO (Token-based) with httpOnly cookie + localStorage fallback.
    - Protection: Strict CSRF tokens, CORS origin filtering, and Helmet.js headers.
- Payments: Razorpay (Dynamic SDK loading, Webhook handlers, live-mode ready).
- Infrastructure: 
    - Image Hosting: Cloudinary (Auto-optimization, transformations).
    - Email: Resend SDK (Transactional emails for orders/auth).
    - Database: PostgreSQL (Managed on Render/Supabase).

3. DATABASE ARCHITECTURE (PRISMA)
---------------------------------
- Polymorphic Relationship Pattern:
    The 'Payment' model supports multiple order types (Physical 'Order' vs. 'ServiceOrder') 
    using 'internalOrderId' and 'orderType'. Note: No strict DB-level FK constraints are 
    used on these polymorphic fields to ensure compatibility with high-concurrency 
    cloud environments like Render.
- Role-Based Access Control (RBAC):
    Users are categorized into [CUSTOMER, ADMIN, SUPER_ADMIN]. Permissions are enforced 
    at both the API middleware and Frontend routing layers.
- Printing Service Schema:
    ServiceOrder model handles complex customization metadata (URLs, text instructions) 
    tailored for printing-specific workflows.

4. DESIGN SYSTEM & UI STANDARDS
-------------------------------
- Aesthetic Identity: 
    "Luxury Minimalist" — Using HSL-tailored colors (Brand Primary: #5a3f2f, Secondary: #b88a2f), 
    custom glassmorphism, and premium typography (Inter/Outfit).
- Mobile-First Responsiveness:
    All critical Purchasing and Tracking pages (Cart, Checkout, Order Detail) are 
    standardized to prevent text truncation and layout overflows on small viewports.
- Standard Components:
    - Custom Scrollbars: Consistent luxury grain scrollbars across all modules.
    - Status Badges: Color-coded progress indicators (e.g., bg-brand-surface for Pending).
    - Premium Modals: Backdrop blur and spring-based animations.

5. AUTHENTICATION & SESSION HYDRATION
-------------------------------------
- Multi-Layered Persistence:
    Ensures seamless sessions across standard browsers and restrictive environments 
    (Safari/In-App) by synchronizing httpOnly cookies with localStorage.
- Interceptor Strategy:
    Axios interceptors manage token lifecycle, including:
    - Transparent token refresh on 401 status.
    - CSRF token bootstrapping for unsafe HTTP methods.
    - Graceful logout on refresh-token expiration.

6. CUSTOM PRINTING SERVICE SUITE
--------------------------------
- Promotion Workflow:
    Service requests (Stickers, Letterheads, BillBooks) are created as 'PENDING' 
    until payment is confirmed via Razorpay. Upon 'PAID' status, they are promoted 
    to 'CONFIRMED' and appear in the user's Service History.
- Lead Management:
    Services are handled as "Lead-to-Order" conversions, where administrative 
    review happens after initial payment confirmation.

7. RECENT ARCHITECTURAL EVOLUTIONS
----------------------------------
- Mobile UI Refinement (May 2026): Full audit and refactor of Cart and Checkout modules 
  for mobile responsiveness, including full-width search overlays and stacked summary cards.
- Performance: Implemented lazy-loading for heavy scripts (Razorpay) and 
  image auto-transformation via Cloudinary.
- Code Reliability: Transitioned to atomic Zustand stores and resolved 401 
  interceptor loop bugs to improve hydration stability.

8. DIRECTORY OVERVIEW
---------------------
- /backend/src/api: Core business logic and controllers.

- /frontend-store/src/api: Centralized API client (client.js) and endpoint definitions.
